## 3. Data Flow — Corrected

```
npm install axios
       │
       ▼
.npmrc:  registry=http://localhost:4873
       │
       ▼
┌─────────────────────────────────────────────────────────┐
│ proxy/server.js                                          │
│   ├─ GET /axios               → PACKUMENT path          │
│   │    1. forward to upstream                           │
│   │    2. parse packument JSON                          │
│   │    3. for each version V in packument.versions{}:   │
│   │         baseline = witness.getBaseline(pkg, V)      │
│   │         if baseline:                                 │
│   │           results = gates.run(incoming[V], baseline) │
│   │           decisions[V] = aggregate(results)          │
│   │         else:                                        │
│   │           witness.recordBaseline(pkg, V, incoming[V])│
│   │           decisions[V] = {ALLOW, reason: first-seen} │
│   │    4. rewriter.rewrite(packument, decisions)         │
│   │       · strip BLOCK versions from versions{} + time{}│
│   │       · if latest was blocked, downgrade to highest │
│   │         non-blocked semver + WARN on stderr         │
│   │    5. log decisions to gate_decisions               │
│   │    6. print WARN lines to stderr for legit versions │
│   │    7. return modified packument (HTTP 200)          │
│   │                                                      │
│   └─ GET /axios/-/axios-1.14.1.tgz → TARBALL path       │
│        1. parse URL → (pkg, V)                          │
│        2. check overrides table                          │
│        3. read decision from gate_decisions              │
│        4. if BLOCK and not overridden → HTTP 403 + JSON  │
│        5. else forward to upstream, stream tarball       │
└─────────────────────────────────────────────────────────┘
```

**Why packument-level, not install-level**: npm fetches the full
packument once and picks a version from its own resolver. A 403 on the
packument fails *every* install of that package, including legitimate
old versions. Rewriting lets npm's resolver never see the blocked
versions in the first place. This is the Verdaccio / Snyk pattern.

**Why the tarball path also matters**: if the user pins an exact
version (`npm install axios@1.14.1`) npm may bypass the packument and
go straight to the tarball URL. The tarball handler is the safety net.

**`dist-tags.latest` downgrade rule**:
- If `dist-tags.latest === V_blocked`: walk `versions{}` in semver-desc
  order, pick the highest version whose decision is `ALLOW` or `WARN`,
  set `dist-tags.latest = that_version`, set
  `dist-tags.chaingate-previous-latest = V_blocked` for observability.
- Print a loud stderr banner: `⚠  chaingate: latest downgraded
  {pkg}@{V_blocked} → {V_new} (N gate violations)`.
- If *every* version is BLOCKED: leave the packument empty and let
  npm fail naturally with "no matching version". Log it.

## 4. SQLite Schema (V1, npm-only)

```sql
-- Packages ---------------------------------------------------------
CREATE TABLE packages (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    ecosystem    TEXT NOT NULL CHECK (ecosystem = 'npm'),  -- V1: npm only
    package_name TEXT NOT NULL,
    UNIQUE (ecosystem, package_name)
);

-- Versions ---------------------------------------------------------
CREATE TABLE versions (
    id                          INTEGER PRIMARY KEY AUTOINCREMENT,
    package_id                  INTEGER NOT NULL REFERENCES packages(id),
    version                     TEXT NOT NULL,
    published_at                TEXT,                -- ISO 8601
    -- content
    content_hash                TEXT,                -- dist.shasum (sha1, legacy npm)
    content_hash_algo           TEXT,
    integrity_hash              TEXT,                -- dist.integrity SRI (sha512-...)
    git_head                    TEXT,                -- gitHead when present
    package_size_bytes          INTEGER,
    -- deps (npm groups)
    dependency_count            INTEGER,
    dependencies                TEXT,                -- JSON string
    dev_dependencies            TEXT,                -- JSON string
    peer_dependencies           TEXT,                -- JSON string
    optional_dependencies       TEXT,                -- JSON string
    bundled_dependencies        TEXT,                -- JSON string
    dev_dependency_count        INTEGER,
    peer_dependency_count       INTEGER,
    optional_dependency_count   INTEGER,
    bundled_dependency_count    INTEGER,
    -- publisher
    publisher_name              TEXT,                -- _npmUser.name
    publisher_email             TEXT,                -- _npmUser.email
    publisher_tool              TEXT,                -- _npmVersion
    maintainers                 TEXT,                -- JSON: maintainers[]
    -- provenance
    publish_method              TEXT,                -- 'oidc' | 'token' | 'unknown'
    provenance_present          INTEGER,             -- 0/1 (SQLite bool)
    provenance_details          TEXT,                -- JSON
    -- scripts (Gate 6 input)
    has_install_scripts         INTEGER,             -- 0/1; true if scripts.{pre,}install or postinstall
    -- metadata
    source_repo_url             TEXT,                -- repository.url
    license                     TEXT,                -- license field (SPDX string)
    -- lifecycle
    first_observed_at           TEXT DEFAULT (datetime('now')),
    last_seen_at                TEXT,
    UNIQUE (package_id, version)
);

-- Version files (one row per tarball in V1 — kept for parity with VPS schema)
CREATE TABLE version_files (
    id                INTEGER PRIMARY KEY AUTOINCREMENT,
    version_id        INTEGER NOT NULL REFERENCES versions(id),
    filename          TEXT NOT NULL,
    packagetype       TEXT,                          -- 'tarball' for npm
    content_hash      TEXT,
    content_hash_algo TEXT,
    size_bytes        INTEGER,
    uploaded_at       TEXT,
    url               TEXT,
    first_observed_at TEXT DEFAULT (datetime('now')),
    last_seen_at      TEXT,
    UNIQUE (version_id, filename)
);

-- Gate decisions (per-user, starts empty on fresh init) ------------
CREATE TABLE gate_decisions (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    package_name TEXT NOT NULL,
    version      TEXT NOT NULL,
    disposition  TEXT NOT NULL CHECK (disposition IN ('ALLOW','WARN','BLOCK')),
    gates_fired  TEXT NOT NULL,                      -- JSON: [{gate,result,detail}]
    decided_at   TEXT DEFAULT (datetime('now'))
);

-- Overrides --------------------------------------------------------
CREATE TABLE overrides (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    package_name TEXT NOT NULL,
    version      TEXT NOT NULL,
    reason       TEXT NOT NULL,
    created_at   TEXT DEFAULT (datetime('now')),
    UNIQUE (package_name, version)
);

-- Seed provenance (populated by export_seed.py, verified on import)-
CREATE TABLE seed_metadata (
    key          TEXT PRIMARY KEY,
    value        TEXT NOT NULL
);
-- Rows: schema_version, exported_at, source_host, row_counts,
--       signing_key_fingerprint, seed_version

-- Indexes ----------------------------------------------------------
CREATE INDEX idx_versions_pkg       ON versions(package_id);
CREATE INDEX idx_versions_pub_at    ON versions(package_id, published_at DESC);
CREATE INDEX idx_vfiles_ver         ON version_files(version_id);
CREATE INDEX idx_decisions_pkg_ver  ON gate_decisions(package_name, version);
CREATE INDEX idx_decisions_at       ON gate_decisions(decided_at DESC);
CREATE UNIQUE INDEX idx_overrides   ON overrides(package_name, version);
```

**Columns explicitly dropped from P5_updated.md's schema (pypi-only):**
`publisher_maintainer`, `publisher_maintainer_email`, `license_text`,
`license_expression`. The V1 `license` column is the SPDX string that
npm exposes in the packument.

**Append-only invariant (ported from VPS):**
- `versions` and `version_files` are append-only. `recordBaseline` uses
  `INSERT OR IGNORE`. Mutations happen only through write-once helpers
  (`bumpLastSeen`), never through bare `UPDATE`.
- `gate_decisions` is append-only (every decision is a new row).
- `overrides` is the only table with free-form updates (by user action).

## 5. Seed Export + Ed25519 Signing

### 5.1 Export process (`collector/export_seed.py`)

Runs on the VPS, reads Postgres, writes a signed artifact bundle.

```
1. Open Postgres read-only connection to `chaingate`.
2. Create fresh chaingate-seed.db SQLite file (overwrite if exists).
3. CREATE SCHEMA (same DDL as §4).
4. COPY packages (only ecosystem='npm' in V1).
5. COPY versions (drop raw_metadata column). Map Postgres bool → SQLite 0/1.
6. COPY version_files (drop raw_metadata, drop attestation_* columns — V1 doesn't ship those).
7. INSERT seed_metadata rows: schema_version, exported_at, source_host,
   row_counts (packages/versions/files), signing_key_fingerprint, seed_version.
8. Run VACUUM + PRAGMA optimize. Close.
9. Compute sha256(chaingate-seed.db) → write chaingate-seed.db.sha256 (hex string).
10. Sign sha256 bytes with Ed25519 privkey → write chaingate-seed.db.sig (raw 64 bytes).
11. Write chaingate-seed.db.manifest.json:
    {
      "seed_version": "2026.04.14.1",
      "exported_at": "2026-04-14T...",
      "row_counts": { "packages": 104, "versions": 50423, "version_files": 50423 },
      "sha256": "<hex>",
      "signing_key_fingerprint": "<first 16 hex of pubkey>",
      "schema_version": 1
    }
12. Upload all four files to GitHub Release asset (manual step for v0.1;
    automated in v1.1).
```

**Privkey custody:**
- Ed25519 keypair generated once on VPS: `openssl genpkey -algorithm Ed25519 -out /root/chaingate/signing/privkey.pem && chmod 0400 /root/chaingate/signing/privkey.pem`
- Privkey never leaves the VPS. No backups in cloud storage. Manual
  paper-backup of the seed phrase (optional).
- Pubkey extracted as raw 32-byte Ed25519 pubkey, base64-encoded, and
  hard-coded as a literal constant in `witness/seed_verify.js`.

### 5.2 Verification (`witness/seed_verify.js`)

On every `scw init` or `scw update-seed`:

```javascript
import { verify } from 'node:crypto';

// Embedded literal — the only trust anchor in the CLI
const CHAINGATE_SEED_PUBKEY_B64 = 'MCowBQYDK2VwAyEA...';  // filled at release
const CHAINGATE_SEED_PUBKEY = /* decoded 32 bytes */;

export async function verifySeed(dbPath, sha256Path, sigPath) {
  // 1. Read the three files
  const localHash = sha256File(dbPath);              // hex string
  const claimedHash = fs.readFileSync(sha256Path, 'utf8').trim();
  const sig = fs.readFileSync(sigPath);              // 64 raw bytes

  // 2. Integrity check
  if (localHash !== claimedHash) {
    throw new Error(`seed hash mismatch: local=${localHash} claimed=${claimedHash}`);
  }

  // 3. Signature check — sign target is the hex sha256 string
  const ok = verify(null, Buffer.from(claimedHash), {
    key: CHAINGATE_SEED_PUBKEY,
    format: 'der',
    type: 'spki',
  }, sig);
  if (!ok) {
    throw new Error('seed signature verification failed');
  }
}
```

**Threats addressed:**
- GitHub Release tampering → detected (sig fails).
- MITM on download → detected (sig fails).
- CDN cache poisoning → detected (sig fails).
- Offline-only users → can bring `.db + .sha256 + .sig` via USB;
  `scw init --seed /path/to/dir` verifies locally.

**Threats NOT addressed (documented, not mitigated in V1):**
- VPS signing-key compromise → need to rotate pubkey, cut a new CLI release.
- Compromise of the `chaingate` npm publish itself → attacker could ship
  a CLI with their own embedded pubkey. **Mitigation**: chaingate's own
  publish pipeline uses `npm publish --provenance` via GitHub Actions
  with OIDC. Consumers can `npm audit signatures` to verify that the
  `chaingate` tarball they installed was built by the real GitHub
  workflow. This closes the loop: our threat chain bottoms out at
  GitHub Actions' OIDC trust, which is also what npm's native
  provenance relies on.

### 5.3 Air-gap support

`scw init --no-seed`: skips seed download entirely. User gets an empty
`witness.db` and the proxy runs in pure learn-on-observe mode.

`scw init --seed /path/to/chaingate-seed.db`: expects `.sha256` and
`.sig` as siblings; verifies before importing. Intended for users
behind firewalls that can't hit GitHub.

## 6. The Six Gates (Corrected Specs)

Each gate is a pure function:

```typescript
type GateInput = {
  ecosystem: 'npm';
  packageName: string;
  version: string;
  incoming: VersionMetadata;      // from upstream packument
  baseline: VersionMetadata|null; // from witness.db
  history: VersionMetadata[];     // all prior versions, newest first
  config: GateConfig;
};
type GateResult = {
  gate: string;
  result: 'ALLOW' | 'SKIP' | 'WARN' | 'BLOCK';
  detail: string;
};
```

`SKIP` is new vs P5_updated.md: it means "this gate had no signal to
evaluate" (first-seen version, missing baseline field, etc.). Does not
count toward the WARN threshold.

### Gate 1: content-hash (BLOCK) — NARROWED

```
PURPOSE: detect bytes-under-same-version-string replacement.
         Answers exactly one question: "has the tarball under
         (pkg, version) we already observed been replaced?"

LOGIC:
  if baseline is null:
    return SKIP  // first-seen; nothing to compare against
  if baseline.integrity_hash != incoming.integrity_hash
     OR baseline.content_hash != incoming.content_hash:
    return BLOCK (`integrity hash differs from baseline: ${old} → ${new}`)
  return ALLOW

DOES NOT FIRE on: new version strings. A version we've never seen before
                  is handled by gates 2-6, not this one.

FALSE POSITIVES: Zero by construction. Immutable bytes under an
                 immutable version string is the entire promise of
                 a package registry.

REAL-WORLD MATCH: Trivy-style re-publish attacks, registry takeover
                  campaigns, single-version tarball swaps.
```

### Gate 2: dep-structure (WARN)

```
PURPOSE: detect new dependencies appearing in a version bump.

LOGIC:
  if history.length == 0:
    return SKIP
  prior = union of dependencies{} across all history (dependencies only,
          NOT dev/peer/optional — those are build-time and low-risk)
  new_deps = incoming.dependencies.keys - prior
  if new_deps is empty:
    return ALLOW
  return WARN(`new dependency: ${new_deps[0]} (not in prior ${history.length} versions)`)

FALSE POSITIVES: Medium. Legitimate refactors add deps.
```

### Gate 3: publisher-identity (WARN)

```
PURPOSE: detect maintainer handoff or account takeover.

LOGIC:
  if baseline is null:
    return SKIP
  latest_prior = history[0]
  if latest_prior.publisher_email is null:
    return SKIP  // data gap, not anomaly
  if latest_prior.publisher_email == incoming.publisher_email:
    return ALLOW
  return WARN(`publisher changed: ${old} → ${new}`)

FALSE POSITIVES: Medium. Real maintainer changes happen.
```

### Gate 4: provenance-continuity (WARN)

```
PURPOSE: detect loss of OIDC attestation in a previously-attested package.

LOGIC:
  if history.length < 2:
    return SKIP  // need at least two prior observations
  any_prior_had_provenance = any(h.provenance_present == 1 for h in history)
  if not any_prior_had_provenance:
    return SKIP  // package has never used Trusted Publishing
  if incoming.provenance_present == 1:
    return ALLOW
  return WARN(`OIDC provenance missing (${N} prior versions had it)`)

FALSE POSITIVES: Low. Legitimate CI moves that drop OIDC are rare.
```

### Gate 5: release-age (WARN)

```
PURPOSE: configurable cooldown on freshly-published versions.

LOGIC:
  age_hours = (now - incoming.published_at) / 3600
  if age_hours >= config.release_age_hours (default 72):
    return ALLOW
  return WARN(`version is ${age_hours}h old (threshold ${config.release_age_hours}h)`)

FALSE POSITIVES: Low. Configurable.
NOTE: Commoditized by npm 11.10+'s native min-release-age. Kept for
      completeness, not positioned as a differentiator.
```

### Gate 6: scope-boundary (WARN or BLOCK)

```
PURPOSE: the Axios combo detector. Individually-soft signals combining
         into a high-confidence malicious indicator.

LOGIC:
  new_deps = result from Gate 2
  if new_deps is empty:
    return ALLOW
  if not incoming.has_install_scripts:
    return ALLOW
  // We have (new dep) + (install scripts). That's suspicious.
  // Check if the new dep is itself freshly published.
  for dep in new_deps:
    dep_info = out_of_band_fetch(dep, 24h cache)
    if dep_info is null:  // can't determine
      continue
    dep_age_h = (now - dep_info.first_publish) / 3600
    if dep_age_h < 24:
      return BLOCK(`new dep ${dep} published ${dep_age_h}h ago + install scripts`)
  return WARN(`new dep ${new_deps[0]} + install scripts detected`)

OUT-OF-BAND FETCH: /proxy/registry fetches the new dep's packument once,
                   caches in memory for 24h. At most one extra request
                   per novel dep. Rate-limited to 5 rps total.

FALSE POSITIVES: WARN variant: medium. BLOCK variant: very low — all three
                 signals aligning is rare outside actual attacks.
```

### Gate runner — `gates/index.js`

```
Run all six gates in parallel. Collect GateResults[].
If overrides table has a row for (pkg, version):
  disposition = ALLOW (with note: "override: {reason}")
Else:
  blocks = count(r.result == 'BLOCK')
  warns  = count(r.result == 'WARN')
  if blocks > 0:
    disposition = BLOCK
  elif warns >= config.warn_escalation_threshold (default 4):
    disposition = BLOCK  (escalated)
  elif warns > 0:
    disposition = WARN
  else:
    disposition = ALLOW

Always log to gate_decisions with:
  { package_name, version, disposition, gates_fired: GateResults[] }

Return decision object up to the rewriter.
```

## 8. CLI Commands

### `scw init [--no-seed] [--seed <path>] [--scope user|project]`

```
1. Create ~/.chaingate/ (or ./.chaingate/ when --scope project)
2. Seed handling:
   - Default: download seed.db + sha256 + sig from pinned GH Release URL,
     verify via witness/seed_verify.js, copy to ~/.chaingate/witness.db
   - --no-seed: create empty witness.db with schema only
   - --seed <path>: expect .sha256 and .sig siblings, verify locally
3. Write config.json (defaults per §7)
4. Detect .npmrc location (user or project per --scope)
5. Back up original .npmrc → ~/.chaingate/npmrc.backup
6. Append `registry=http://localhost:4873` to .npmrc
7. Spawn proxy as detached background process, PID in ~/.chaingate/proxy.pid
8. Print: "Ready. N packages, M versions in witness store. Proxy: localhost:4873"
```

### `scw status`

Prints a table:
```
Witness store:   104 packages, 50,423 versions, 50,423 files
Seed version:    2026.04.14.1 (exported 2026-04-14, verified OK)
Proxy:           running on localhost:4873 (pid 12345)
Upstream:        https://registry.npmjs.org
Decisions:       12,847 total · 12,801 ALLOW · 41 WARN · 5 BLOCK
Last 5 decisions:
  axios@1.7.9       ALLOW  2026-04-14 18:32:11
  axios@1.14.1      BLOCK  2026-04-14 18:32:07 (scope-boundary, 3 WARN)
  ...
```

### `scw allow <pkg>@<version> --reason "..."`

```
1. Validate pkg@version syntax
2. INSERT OR REPLACE INTO overrides (package_name, version, reason) VALUES (...)
3. Print: "Override recorded. {pkg}@{version} will be allowed on next install."
4. Live effect — no restart needed, proxy reads overrides on every gate run.
```

### `scw stop`

```
1. Send SIGTERM to proxy PID in ~/.chaingate/proxy.pid
2. Restore ~/.chaingate/npmrc.backup → original .npmrc location
3. Remove proxy.pid
4. Print: ".npmrc restored. Proxy stopped."
```

### `scw update-seed`

```
1. Fetch latest seed manifest from pinned GH Release URL
2. Compare manifest.seed_version with seed_metadata.seed_version in witness.db
3. If same: print "Already up to date." and exit
4. If newer: download + verify + atomic swap ~/.chaingate/witness.db
5. Preserve gate_decisions and overrides tables across swap
6. Print row-count delta
```

## 9. Write-on-Observe (unchanged from P5_updated.md §9)

When the proxy sees `(pkg, V)` for the first time (no baseline row),
it writes one. The fields populated match what the VPS collector
writes, but sourced from the live packument. Second install of the
same `(pkg, V)` triggers gate comparison.

After ~1 week of normal use, `witness.db` covers the user's full
dependency tree. Seed data gives day-one protection for the 209
top packages; learn-on-observe handles the long tail.

## 13. Dependencies (User-Facing)

```json
{
  "engines": { "node": ">=22.0.0" },
  "dependencies": {
    "better-sqlite3": "^11.0.0",
    "undici": "^6.0.0",
    "semver": "^7.6.0"
  },
  "devDependencies": {
    "node:test": "built-in"
  }
}
```

Three production deps:
- **better-sqlite3** — synchronous SQLite. Prebuilt binaries for
  linux-x64, linux-arm64, darwin-x64, darwin-arm64, win32-x64. Alpine/
  musl users must compile; README notes this.
- **undici** — Node's reference HTTP client. ~40 lines of dispatcher
  code for the two verbs we need (GET packument, GET tarball).
- **semver** — used by the rewriter to pick "highest non-blocked
  version" when downgrading `dist-tags.latest`. npm's own semver lib.

No Python. No test-framework dep (node:test is built-in). No
cryptography library (Ed25519 verification uses node:crypto).

## 16. Competitive Positioning

ChainGate is a **witness**, not a firewall. It does not compete with
any tool that does behavioral analysis, pattern matching, or external
threat intel. It detects one class of anomaly: **metadata divergence
from local history**. That is the entire product.

| Tool              | What they do                       | Where ChainGate fits                   |
|-------------------|-------------------------------------|----------------------------------------|
| Socket Firewall   | 70+ behavioral signals, cloud API   | Local-only metadata diff; no cloud     |
| Datadog SCFW      | Blocks known-malicious (OSV feed)   | Detects unknown via historical baseline |
| npm min-release-age | Cooldown timer on new versions    | Release-age is 1/6 gates, not the core |
| Go sumdb          | Content-hash log, Go-only           | Same concept, npm + PyPI + self-hosted |

The five gaps V1 fills:
1. Cross-ecosystem content-hash verification against a historical baseline
2. Append-only transparency of package metadata for npm (PyPI in M5-6)
3. Deterministic structural gating without external intelligence
4. Self-hosted, air-gapped, zero-cloud supply-chain gating
5. Compliance-mappable enforcement for regulated environments (NCA ECC / SAMA CSF)
